# Full-project audit & constraints reference

Generated by an adversarial multi-agent audit (41 agents, 94 endpoints across 9 domains
mapped, 25 raw findings → **23 confirmed real** after refutation passes). Live API behavior
was separately verified end-to-end against the production server with the demo vendor
(50/50 checks green). The two bugs found by live testing — the dedup `storageKey` 500 and
the temporary-link missing-`expiresIn` guard — are already fixed (v1.0.18).

The full live API reference is also available in-app at **`/docs`** and machine-readable at
**`GET /api/v1/openapi`**.

## Severity summary

| Severity | Count | Theme |
|---|---:|---|
| High | 8 | API-key bucket scoping bypass, 2× ReDoS, maxDownloads race, private-link JWT scope, multipart size/quota bypass, quota-accounting drift, folder-move path corruption |
| Medium | 9 | suspended-vendor mutations, temp-link never-expire, link password in URL, restore quota, per-file size cap, zip error hang, edit orphans object, orphan-sweep multipart leak, multipart init validation |
| Low | 6 | JWT-revocation scoping/TTL, count-limit races, usage underflow |

None are cross-tenant breaches — every data query is correctly constrained by `vendorId`.
The high-severity issues are **within-tenant** authorization/integrity bugs (cross-bucket
within a vendor, quota/billing integrity, DoS).

## HIGH

### H1 — API-key `bucketId` scoping is bypassed on per-file endpoints
**Files:** `app/api/v1/files/[id]/**` (download, route DELETE/PATCH, content, copy, extract, links, links/reset, restore, hide/unhide), `lib/rbac.ts:138-140`
`can()` only enforces an API key's `bucketIds` restriction when the caller passes a `bucketId`
in the resource arg. Per-file routes call `can()` with `{vendorId}` only, then load the file by
`{_id, vendorId}` — so a key scoped to bucket A can read/download/overwrite/delete/link files in
**any** of the vendor's buckets. PATCH (rename/move/retag) performs no `can()` at all beyond the
vendorId filter. The mechanism works where used (e.g. `GET /files/:id` passes `bucketId`), so the
vulnerable routes simply fail to use it.
**Fix:** load the file first and pass `{vendorId, bucketId: String(file.bucketId)}` to `can()`; or make
`can()` fail-closed when `principal.bucketIds` is non-empty but no `resource.bucketId` was supplied.

### H2 — ReDoS / RegExp injection in file search (`q`)
**File:** `app/api/v1/buckets/[bid]/files/route.ts:40-51`
`q` is read raw and compiled into `new RegExp(q, 'i')` with no length cap or metacharacter
escaping, then run by MongoDB over `originalName` for every candidate doc. A catastrophic pattern
(`(a+)+$`) pins DB CPU; `.*`/`^` broadens matches into a filename oracle. No rate limiting exists
(`lib/ratelimit.ts` is defined but never called).
**Fix:** cap length + escape metacharacters (`q.replace(/[.*+?^${}()|[\]\\]/g, '\\$&')`), or use the
existing `$text` index (`models/File.ts:63`).

### H3 — ReDoS / RegExp injection in admin vendor search (`q`)
**File:** `app/api/v1/admin/vendors/route.ts:25-29`
Same bug class: `q` → `new RegExp(q,'i')` over `name`/`slug` across the whole Vendor collection,
reachable by `platform_staff`. Self-inflicted DoS.
**Fix:** same as H2.

### H4 — `maxDownloads` enforcement is racy (TOCTOU)
**File:** `lib/link-handler.ts:39-58, 127`
The count is read at request start (`isLinkUsable`) and `$inc`-incremented unconditionally at the
end. N concurrent requests on a `maxDownloads:1` link all read `0`, all pass, all serve the file.
**Fix:** gate on an atomic `findOneAndUpdate({_id, status:'active', $expr:{$lt:['$downloadCount','$maxDownloads']}}, {$inc:{downloadCount:1}})` and only serve when it matched.

### H5 — Private-link JWT `bucketIds` scope is never enforced
**Files:** `lib/links.ts:58-63`, `lib/link-handler.ts:101-117`
3rd-party JWTs carry a `bucketIds` claim that is enforced for API keys and transfer tokens, but
the private-link path checks only `vendorId` + `scopes`. Any vendor JWT with the generic
`file:download` scope can download **every** private link of that vendor, including buckets the
JWT was explicitly scoped away from — a vendor-wide download master key.
**Fix:** after loading the file in `handleLinkDownload`, reject when `jwt.bucketIds` is non-empty and
does not include `String(file.bucketId)`.

### H6 — Multipart `complete` trusts client `sizeBytes`; skips quota/size/suspended
**File:** `app/api/v1/files/multipart/[uploadId]/complete/route.ts:36-47`
Accounted size is `body.sizeBytes ?? draft.sizeBytes` (client-controlled). No `checkQuota`, no
`MAX_UPLOAD_BYTES`, no bucket `maxFileSizeBytes`, no suspended check, and `completeMultipart()`
issues no `HeadObject`. `init` also only checks size when `sizeBytes` is truthy. A client can upload
gigabytes across parts then `complete` with `sizeBytes:1`, bypassing quota and under-reporting usage;
suspended vendors can also upload via this path.
**Fix:** `HeadObject` after completion for the authoritative `ContentLength`, then run
quota/size/suspended checks at complete time (pattern already exists in `putObjectStream`).

### H7 — Quota accounting drifts negative across trash → recount → purge
**Files:** `app/api/v1/files/[id]/route.ts:78-96`, `app/api/internal/cron/recount-usage/route.ts`, `purge-trash/route.ts:42`
DELETE only sets `status:'trashed'` (no `decrementUsage`); `recount-usage` aggregates only
`status:'ready'` (silently subtracting trashed bytes, handing back quota); `purge-trash` then
`decrementUsage`s the same file again → `usage.storageBytes`/`fileCount` go negative, and
`checkQuota` against a negative base grants effectively unlimited extra quota.
**Fix:** make both reconcilers agree — either count trashed files in the recount aggregation, or
`decrementUsage` at trash time; and clamp decrements at 0.

### H8 — Folder move leaves descendant `path` values stale
**File:** `app/api/v1/folders/[id]/route.ts:35-52`
On move, only the moved folder's `path` is recomputed; children/grandchildren keep their old
`path`. Subtree logic relies on path prefixes: `archive` (`buckets/[bid]/archive/route.ts:53-56`)
silently omits descendant files, and `transfer/manifest` rebuilds wrong folder paths.
**Fix:** bulk-update every descendant's `path` (rewrite the old-prefix → new-prefix) in the same op.

## MEDIUM

### M1 — Suspended vendors can still delete buckets/folders and hide/unhide
**Files:** `buckets/[bid]/route.ts` (DELETE/PATCH), `folders/[id]/route.ts` (DELETE/PATCH), `folders|files/[id]/hide,unhide`
`SUSPENDED_BLOCKED` only covers `file:upload|download|delete` + `publicurl:create`. Bucket/folder
mutations use other permissions and these routes have no explicit suspended check, so a suspended
vendor can delete (empty) buckets, delete/rename/move folders, and toggle visibility — contradicting
the "suspended actions are inert" intent. (Destruction is bounded: DELETE requires the container be empty.)
**Fix:** add `if (p.vendorStatus==='suspended') return suspended();` to these handlers, or extend `SUSPENDED_BLOCKED`.

### M2 — `temporary` links can be made to never expire via `neverExpire`
**Files:** `lib/links.ts:44`, `app/api/v1/files/[id]/links/route.ts`
`{type:'temporary', expiresIn:3600, neverExpire:true}` passes the expiresIn guard but `clampExpiresIn`
returns `null` for `neverExpire`, producing a permanent "temporary" link the expiry cron never touches.
**Fix:** reject `neverExpire` when `type==='temporary'` (schema or route), or ignore it for temporary.
*(Adjacent to the v1.0.18 expiresIn fix — small follow-up.)*

### M3 — Link password accepted via `?p=` query string
**File:** `lib/link-handler.ts:96`
Secret-link passwords are accepted from the URL query in addition to the header. Query strings leak
into proxy/CDN logs, browser history, and the `Referer` header (CWE-598). Verify logic itself is fine.
**Fix:** accept the password only via header/body, not the URL.

### M4 — Restore from trash skips quota re-check (and suspended)
**File:** `app/api/v1/files/[id]/restore/route.ts:17-22`
Flips `trashed→ready` with no `checkQuota`/suspended check. Combined with the recount behavior (H7) or
a plan downgrade, a vendor can push `usage` above `maxStorageBytes`.
**Fix:** `checkQuota(vendorId, file.sizeBytes)` → `quotaExceeded()`, and honor suspended.

### M5 — Per-vendor `maxFileSizeBytes` not enforced on inline content edit
**File:** `app/api/v1/files/[id]/content/route.ts:65-69`
Edit calls `checkQuota` with the *delta*, not the new total, so repeated sub-limit edits grow a file
past the per-vendor `maxFileSizeBytes` (bounded by the 5 MB content cap). (Extract/archive use the
aggregate, which is actually a stronger gate — those sub-claims are overstated.)
**Fix:** validate each created/edited file's own `sizeBytes` against the vendor + bucket limit.

### M6 — `executeFsOp` zip: dest WriteStream errors unhandled → request hang
**File:** `lib/fs-ops.ts:72-85`
Only `archive.on('error')` is wired. If the destination `createWriteStream(dest)` errors
(EACCES/ENOSPC/EISDIR/ENOENT), `'close'` never fires → the Promise never settles → request hangs to
`maxDuration` (~300s), and the unhandled stream `'error'` can crash the process.
**Fix:** add `output.on('error', reject)` and `.catch(reject)` on the finalize chain; reject on empty `items`.

### M7 — Inline text edit orphans the previous storage object
**File:** `app/api/v1/files/[id]/content/route.ts:72-86`
PUT writes a new `v<ts>-` key and reassigns `storageKey` but never deletes the old object. For a normal
(non-shared) file every edit leaks one object; no sweep reclaims it.
**Fix:** after save, delete the previous key when no other `File` row references it (same refcount as purge-trash).

### M8 — `orphan-sweep` marks stale uploads `failed` but never aborts the S3 multipart
**File:** `app/api/internal/cron/orphan-sweep/route.ts:19-30`
Drafts stuck `uploading` >24h are flipped to `failed`, but the S3/MinIO multipart upload
(`metadata.uploadId`) is never aborted, so uploaded parts persist forever (no lifecycle rule set).
**Fix:** best-effort `storage.abortMultipart(storageKey, uploadId)` before marking failed.

### M9 — Multipart `init` body is not schema-validated
**File:** `app/api/v1/files/multipart/init/route.ts:30-78`
Only a shallow truthiness check; raw `bucketId/folderId/sizeBytes/originalName/mimeType` flow into
queries and persistence. A malformed `bucketId` throws a Mongoose CastError → 500 instead of 400.
(Tenant-safe — every query is `vendorId`-scoped.)
**Fix:** add a zod schema and `safeParse` like the sibling upload routes.

## LOW

### L1 — JWT revocation not tenant-scoped; no-op upsert
`app/api/v1/jwt/revoke/route.ts:25-36` upserts keyed only by `jti` with `$setOnInsert`; the
download-time check (`lib/link-handler.ts:114`) lacks `vendorId`. `jti` is 128-bit random so
cross-tenant exploitation is impractical, but re-revoke is silently idempotent.
**Fix:** scope revoke filter + lookup by `vendorId`; use `$set` on update.

### L2 — Count limits (`maxBuckets`/`maxApiKeys`) are non-atomic count-then-create
`app/api/v1/buckets/route.ts:41-58`, `api-keys/route.ts:43-46` — concurrent POSTs can overshoot.
Same TOCTOU applies to storage quota (check then increment).
**Fix:** atomic conditional `$inc`/`findOneAndUpdate` or a transaction. *(Bucket name uniqueness is
actually safe — there is a unique index `{vendorId,name}`.)*

### L3 — Usage counters can go negative (no clamp)
`lib/quota.ts:29-34` — `decrementUsage` `$inc`s negatives with no floor (see H7). Drives
`usage.storageBytes`/`fileCount` negative; dashboard %/warning math misbehaves. Self-heals on next recount.
**Fix:** clamp at 0 (`$max` pipeline update) and/or treat negative usage as 0 in `checkQuota`.

### L4 — `JwtRevocation` has `expiresAt` but no TTL index
`models/JwtRevocation.ts:9-12` — unlike `PasswordReset`/`TransferToken`, no TTL index, so revocation
rows accumulate forever and the per-download lookup grows unbounded. (The revoke upsert also never
sets `expiresAt`, so a TTL alone won't purge — the route must set it from the JWT `exp`.)
**Fix:** add `index({expiresAt:1},{expireAfterSeconds:0})` and populate `expiresAt` on revoke.

## Constraints & limits at a glance

### Quota / vendor limits

| Limit | Default | Where |
|---|---|---|
| `Vendor.limits.maxStorageBytes` | per plan | `checkQuota` (`usage.storageBytes + size > max` → 413) |
| `Vendor.limits.maxBuckets` | per plan | `POST /buckets` (`count >= max` → 400) |
| `Vendor.limits.maxApiKeys` | 10 | `POST /api-keys` (active `count >= max` → 400) |
| `Vendor.limits.maxFileSizeBytes` | 500 MiB | `checkQuota` per-file cap; **not** enforced by FS routes/migration |
| `checkQuota` rejection (413 `QUOTA_EXCEEDED`) | — | vendor missing, `status!=='active'`, `size>maxFileSizeBytes`, or storage over |

### Upload / file-size limits

| Constraint | Value | Where |
|---|---|---|
| `env.MAX_UPLOAD_BYTES` | 500 MB | object-storage single upload; multipart init (when `sizeBytes` given) |
| Bucket `settings.maxFileSizeBytes` | `0` = no cap | object-storage upload + multipart init |
| Blank-file `content` | ≤ 5 MB | `POST /buckets/:bid/files/blank` |
| Inline text read | 1 MB (object storage) / 2 MiB (server FS) | `TEXT_MAX` / `readTextFile` |
| Inline text write | ≤ 5 MB (object) / 2 MiB (FS) | `editContentSchema` / `fsOpSchema` |
| Archive (zip) total source | ≤ 1 GB (`SRC_CAP`, in-memory) | `POST /buckets/:bid/archive` |
| Extract source zip | ≤ 500 MB (`ZIP_CAP`) | `POST /files/:id/extract` |
| Extract total uncompressed | ≤ 2 GB (`UNCOMP_CAP`, zip-bomb guard) | `POST /files/:id/extract` |
| Migration s3 per-file | ≤ 256 MiB (larger skipped) | s3 runner |
| Migration bcdnp per-file | no cap; 15-min fetch timeout | bcdnp runner |
| FS uploads / download | **no size/MIME/quota limit** | `/api/v1/fs/*`, `/api/v1/admin/fs/*` |
| Multipart part number | 1..10000 | `PUT /files/multipart/:uploadId/part/:n` |

### Expiry / TTL rules

| Item | Rule | Auto-prune |
|---|---|---|
| Session cookie | `SESSION_TTL_HOURS` (default 12h) | cookie maxAge |
| Impersonation | 60-min TTL | cookie maxAge |
| 3rd-party JWT | `expiresIn` 60s … 1y | `JwtRevocation` (no TTL — see L4) |
| Share link `expiresIn` | 60s … 10y; 1-min floor; `temporary` needs non-null expiry | `isLinkUsable` + lazy `expired` + `expire-urls` cron (no Link TTL index) |
| Link `maxDownloads` | `null` = unlimited; enforced before `$inc` (racy — see H4) | — |
| Password reset | now + 1h, single-use | **TTL index** |
| Transfer token | now + (`hours`‖24)h | **TTL index** |
| API key `expiresAt` | `null` = never | — |
| Trashed files | purge after 30 days | `purge-trash` cron |
| Stale `uploading` | → `failed` after 24h | `orphan-sweep` cron (no multipart abort — see M8) |
| Presigned download URL | 300s | storage backend |

### Suspended-vendor & RBAC notes

- `SUSPENDED_BLOCKED = [file:upload, file:download, file:delete, publicurl:create]`. **Not** blocked
  when suspended: `apikey:create/revoke`, `publicurl:revoke`, `settings:update`, all `admin:*`, and
  bucket/folder mutations (see M1).
- `/auth/login` blocks suspended vendors from a new session (403); pending vendors allowed.
- FS routes use **no `can()`** — gated on `vendorId` + non-suspended (vendor) or `super_admin` (admin).
- Plan/payment-config **writes** need `admin:maintenance:toggle` (super_admin); **reads** need only `admin:vendor:read`.
- Maintenance mode → 503 at `/auth/login` (non-admins) and on all link-download paths.
